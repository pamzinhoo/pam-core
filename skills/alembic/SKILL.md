# alembic

## Purpose
Keep Alembic migrations accurate, reviewable, and aligned with SQLAlchemy models.

## Auto Activation
Use when editing `alembic.ini`, `env.py`, Alembic revision files, autogenerate output, upgrade/downgrade logic, or migration commands.

## Do Not Activate
Do not use for projects with a different migration tool unless Alembic is being adopted explicitly.

## Detect
Look for `alembic`, `versions`, `op.create_table`, `op.add_column`, `revision`, `down_revision`, `upgrade()`, `downgrade()`, and migration heads.

## Responsibilities
- Review autogenerated migrations before accepting them.
- Keep revisions small and tied to intentional schema changes.
- Preserve downgrade behavior when the project requires reversible migrations.
- Handle data migrations carefully and separately when risk is high.
- Verify model metadata is wired correctly in `env.py`.

## Never Do
- Trust autogenerate blindly.
- Drop or rewrite data-bearing columns without an explicit migration plan.
- Leave multiple heads unresolved accidentally.
- Edit applied production migrations without project approval.

## Cooperates With
database-migrations, sqlalchemy, database-design, transactions, testing, code-review.

## Final Checklist
- Revision matches the intended schema change.
- Upgrade and downgrade behavior are deliberate.
- Data-destructive steps are called out.
- Alembic heads and metadata are sane.
- Migration validation ran when practical.

## Examples
- Replace an autogenerated nullable column with a staged migration that backfills before adding `nullable=False`.
- Add a downgrade that drops an index created in the upgrade.
